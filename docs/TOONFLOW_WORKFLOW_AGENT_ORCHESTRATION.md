# Toonflow 工作流程与 Agent 编排方式整理

## 文档目标

本文用于沉淀 Toonflow 的产品工作流与 Agent 编排思路，作为后续在 DeerFlow 中对齐体验的参考基线。

## 信息范围

- 基于 Toonflow-app 对外公开说明与仓库结构可观察信息整理
- 聚焦产品流程与编排模式，不涉及未公开实现细节的断言
- 将“已确认信息”与“工程推断”分层呈现

## 一、工作流程（已确认）

Toonflow 是一条面向短剧/漫剧创作的端到端生产线，主流程为：

1. 角色生成
2. 剧本生成
3. 分镜制作
4. 视频合成

### 1) 角色生成

- 从原始文本中抽取角色
- 生成外貌、性格、身份等结构化设定
- 作为后续剧本与画面生成的上游资产

### 2) 剧本生成

- 基于选定事件/章节进行改编
- 输出对白、场景描述、情节推进等结构化剧本内容

### 3) 分镜制作

- 从剧本自动生成分镜提示词与画面规划
- 包含前中后景、角色动作、道具、场景布局等镜头要素

### 4) 视频合成

- 结合 AI 图像与视频能力生成片段
- 支持在线编辑和个性化输出

## 二、系统模块形态（已确认）

从仓库结构可见，其后端按业务域拆分较明确：

- `src/routes/outline`：大纲域
- `src/routes/script`：剧本域
- `src/routes/storyboard`：分镜域
- `src/routes/video`：视频域
- `src/routes/assets`：资产域
- `src/routes/task`：任务域
- `src/agents`：Agent 模块

该拆分意味着流程阶段已被建模为明确业务边界，而非单一“对话接口”。

## 三、Agent 编排方式（工程推断）

以下为基于产品表现与模块边界的高置信推断：

### 1) 主编排 + 阶段执行

- 存在主编排层负责推进阶段状态
- 各阶段由专门能力单元处理（角色、剧本、分镜、视频）
- 阶段完成后触发下一阶段建议或确认

### 2) 任务化执行与异步状态

- `task` 作为独立业务域，支持长任务运行
- 前端可展示“进行中 / 完成 / 失败”状态
- 用户感知为流程推进，而非底层工具调用细节

### 3) 结果优先的对话风格

- 主对话以阶段结论为主
- 中间过程以状态信息表达，不暴露技术噪音
- 形成“可控进度 + 可交付产物”的创作体验

## 四、体验层关键模式（可复用）

Toonflow 体验的核心并非单点模型能力，而是以下编排协议：

1. 阶段驱动：每次只推动一个明确阶段目标
2. 状态显式：每阶段都有“开始/处理中/完成”信号
3. 产物导向：每阶段必须沉淀可复用产物
4. 低噪输出：主聊天区优先展示业务结论
5. 可继续流：每次完成后给出下一步动作建议

## 五、对 DeerFlow 的映射建议（讨论基线）

### 1) Prompt 层

- 主 Agent 改为“流程编排官”角色
- 子 Agent 改为“阶段产物交付官”角色
- 明确输出协议：状态一句 + 结论一句 + 下一步一句

### 2) 执行层

- 工具调用描述统一业务化文案
- 任务状态统一规范：`in_progress / completed / failed`
- 阶段结果统一落资产，便于复用与回溯

### 3) 展示层

- 主消息显示阶段结论
- 过程步骤默认弱化（折叠或胶囊化）
- 将“技术过程可见性”放到调试层而非主沟通层

## 六、结论

Toonflow 的有效性来自“流程编排能力”而不只是“模型回答能力”：

- 从自由问答转向生产流水线
- 从工具日志转向业务状态
- 从一次性回答转向阶段性交付

这套模式与 DeerFlow 当前能力具备兼容性，可通过 Prompt 约束、任务状态协议和前端呈现策略三层联动实现对齐。
